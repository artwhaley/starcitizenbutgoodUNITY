# Project Rules And Construction Guardrails

This project is aiming toward a shared, persistent space game. Shortcuts that make
one local playtest pass but hide bad scene, prefab, or asset setup are not acceptable.
Prefer stable authored contracts, explicit failures, and testable simulation code.

## Core Rule: Do Not Invent Authored Gameplay Setup

Do not create missing gameplay-critical transforms, nodes, ports, hardpoints, colliders,
entity identities, authority objects, or station/ship attachment points at runtime just
to make a test pass.

Bad pattern:

- A ship is missing `node_docking`, so code creates `new GameObject("node_docking")`.
- A station has no authored docking port, so code creates a default port under the station.
- A weapon has no authored muzzle/hardpoint, so code guesses one from the mesh.
- A prefab is missing a core component, so normal gameplay silently adds it every launch.

Correct pattern:

- Search for the authored object.
- If it is missing, log a clear error naming the expected object and disable that feature.
- Fix the prefab, scene, or imported model intentionally.
- Only add runtime repair behind an explicit temporary path that logs loudly and is ticketed for removal.

Runtime-created transient presentation objects are different. HUD canvases, pooled VFX,
debug text, and projectile visual pools may be created when they are not part of the
authored gameplay contract. They must not define gameplay identity or authoritative poses.

## Hierarchy Rule: Ships And Stations Are Complex Hierarchies

Never assume a ship, station, hardpoint, docking port, collision proxy, or gameplay marker
is a direct child of the root or COG unless the code has a documented reason and test for it.

Ships and stations may be deep hierarchies. Parts may be grouped by mesh, animation rig,
mechanism, imported FBX structure, or future moving assemblies. A docking node may live
under an animated docking-port arm. A hardpoint may live under a turret. A station port may
live several levels below the station root.

When locating authored nodes:

- Search recursively with inactive children included when appropriate.
- Use explicit names, tags, marker components, or serialized references.
- Preserve the authored transform's world pose and local relationship to its actual parent.
- When simulation needs an offset from ship COG/root, compute it relative to that root with
  `root.InverseTransformPoint(node.position)` and `Quaternion.Inverse(root.rotation) * node.rotation`.
- Do not use `node.localPosition` or `node.localRotation` as a COG-relative value unless the node
  is guaranteed to be a direct child of the COG.

## Authored Nodes And Components

Authored transform contracts matter more than convenience.

Docking examples:

- Ship docking node transforms must be authored in the ship hierarchy, named by the documented convention.
- Station docking port transforms must be authored in the station hierarchy.
- Code may attach `ShipDockingNode` or `StationDockingPort` behavior to an existing authored transform.
- Code must not create missing docking transforms.
- If multiple station ports exist, target selection must be explicit or deterministic, and must expose what it selected.

Weapon examples:

- Gun/missile hardpoints must come from authored transforms or explicit component arrays.
- Missing hardpoints should disable firing with a clear error, not spawn shots from a guessed origin.

Collision examples:

- Collision proxies should be authored or generated by a deliberate import/build step.
- Runtime collision behavior should consume explicit collision shapes; it should not infer gameplay collision from arbitrary visual meshes unless that is the documented feature.

## Explicit Wiring Over Silent Repair

Prefab and scene wiring should be visible in the editor. Runtime code should not normalize sloppy prefab setup into something that appears to work.

Acceptable:

- One-time discovery of an already-authored component or transform.
- Clear validation errors that tell the developer exactly what is missing.
- Editor tooling that adds or validates required components as an intentional authoring step.

Avoid:

- Normal-path `AddComponent` for required gameplay components.
- Whole-scene searches every frame.
- Adding default scene roots, ports, nodes, registries, or entities without an explicit authoring reason.
- Fallback behavior that hides which object is actually being used.

## Simulation And Authority Direction

Gameplay simulation should move toward server-authoritative, deterministic-friendly code.

- Keep core simulation code separated from Unity presentation when practical.
- Prefer data inputs and outputs over reading arbitrary scene state during simulation.
- Treat Unity transforms as presentation or authored setup, not as the final authority model.
- When temporary local authority exists, keep the boundary obvious so it can be split out later.
- DTOs, IDs, snapshots, and events should be package-agnostic where possible.

## Diagnostics Must Explain Blocked Behavior

If a feature has several gates, the player/developer should not have to guess which one failed.

For docking capture, log or display enough to distinguish:

- no target port
- ship port not deployed
- station port unavailable
- outside capture distance
- alignment angle too high
- closure speed too high
- moving away from the port
- recapture lockout active

Diagnostics should be throttled when they can fire every frame.

## Tests And Verification

Any fix to simulation math, hierarchy lookup, docking, RCS mapping, weapons, projectiles, collision, or authority flow should include at least one of:

- edit-mode tests for pure math/data behavior
- prefab/scene validation where Unity object setup is the contract
- a manual playtest checklist update
- temporary diagnostic output that can be removed once stable

Build both runtime and editor assemblies before declaring code done:

```powershell
dotnet build Assembly-CSharp.csproj --no-restore
dotnet build Assembly-CSharp-Editor.csproj --no-restore
```

If a build fails from a transient file lock, retry once after a short delay. Do not ignore real compiler errors.

## Agent Rule

When working as an automated coding agent:

- Read the relevant existing code before editing.
- Do not overwrite unrelated user or agent changes.
- Do not replace a file wholesale when a focused patch will do.
- Do not invent missing authored setup.
- Do not assume top-level hierarchy.
- Prefer loud failure over hidden repair.
- Explain any retained fallback and why it is safe.
- Leave the project more inspectable than you found it.
